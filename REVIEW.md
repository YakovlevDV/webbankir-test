### Задание 1 – Code Review

UserApiClient.java

- Зачем класс UserApiClient.java сделан синглтоном, представляю себе архитектуру как просто класс UserApi, 
который содержит методы для работы, стоит обсудить с командой архитектуру этого и подобных классов. Всегда отказывался
от синглтонов в пользу создания обычных объектов, как плюс, это позволяет избежать проблем с распаралелливанием тестов
в будущем.
- Задан таймаут и нигде не используется, тоже стоит обговорить с командой как будем ждать "тяжелые" запросы, чтобы не
получать фейлы, но и не пропустить деградацию API по времени ответа.
- Странно, что метод deleteUser ничего не возвращает, если он не отработает должным образом, то мы даже не 
узнаем об этом
- Совсем нет логирования, .log хотя бы при ошибках нужно делать.
- Нужно обсудить с командой как более удобно передавать Json в параметр метода, через Map<String, Object> очень неудобно. 
Например, можно через JsonNode, добавил Jackson в проект.

BaseTest.java

- System.setProperty("webdriver.chrome.driver", "/usr/local/bin/chromedriver") - выглядит как хардкод путей, нужно вынести
подобные вещи в переменные и передавать как ключи при запуске. Получится запустить и на CI и на локальных машинах. 
Если использовать Selenide нужно будет передать Configuration.browser с нужным браузером
- Не стал бы ограничивать себя созданием конкретной реализации драйвера new ChromeDriver(), скорей всего потребуется 
запуск на Firefox, Safari, Edge. Я бы передавал этот параметр через внешние ключи перед запуском.
- Класс BaseTest, получается общий для API и для UI, тут надо обсуждать. У меня были проекты, где API выполнял прекондишен 
перед UI тестами - тогда ок, если нет то лучше разделелить их.
- @ExtendWith(SeleniumExtension.class) - если правильно понял, то это какой-то самописный класс расширение для настройки свойств 
драйвера и тогда странно, что, например, путь до него прописывается тут. Надо или выносить все в SeleniumExtension
все в этот класс или делать все тут.
- driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10)); - лучше работать с явными ожиданиями, это делает 
тесты более гибкими, когда ждешь чего-то конкретного: появляения/исчезания элемента, например.
- apiClient = UserApiClient.getInstance("http://localhost:8080") - тоже какой-то хардкод, надо выносить в константы если 
не меняется или передавать как ключи через сборщик.
- Еще со своих первых проектов запомнил, чтот лучше проверить драйвер на null, прежде чем закрыть сессию, так не
получишь лишний NPE. Добавил Selenide в проект там делать это вручную нет необходимости.
